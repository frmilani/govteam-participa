# Story E4.1: API research-config — Configuração Pública da Pesquisa

## Status
Done

## Story
**As a** sistema Hub (Formbuilder) que atende milhares de eleitores simultaneamente,
**I want** consultar um endpoint rápido e cacheado do Spoke para saber como renderizar a pesquisa de uma determinada enquete,
**so that** o frontend de pesquisa saiba quais blocos exibir (nps, texto livre, categorias listadas) sem ferir regras de segurança e sem expor informações sigilosas.

## Acceptance Criteria

1. - [x] Criar a rota GET publica `GET /api/public/enquetes/[id]/research-config`.
2. - [x] A rota deve buscar no Prisma a Enquete solicitada, validando se o status permite acesso (ex: "PUBLICADA" ou "EM_ANDAMENTO").
3. - [x] Retornar os campos de configuração da Enquete: `tipoPesquisa`, `modoColeta`, `randomizarOpcoes`, `blocosAdicionais` (parseado de `configPesquisa`), e regras de `distribuicao`.
4. - [x] Fazer include das `categorias` (Segmentos populacionados associados a essa enquete) ordenadas corretamente.
5. - [x] Para cada categoria no payload, retornar a resolução do seu `TemplateQualidade` usando o helper `resolverTemplate(categoriaId)` (já criado na E3.2). Se nulo, omitir.
6. - [x] Segurança (Crucial): A lista de Entidades por categoria NÃO VAI AQUI. A pesquisa via API do formbuilder utilizará rotas autocomplete isoladas para não criar payloads massivos e pesados de 2MB. O research-config envia apenas esqueleto estrutural e templates organizativos (PRD).
7. - [x] Aplicar Cache config do Next.js (ex: `revalidate: 60` segundos) para suportar rajadas de acessos de eleitores lendo a configuração do mesmo banner.
