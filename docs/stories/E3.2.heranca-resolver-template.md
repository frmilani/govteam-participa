# Story E3.2: Herança Hierárquica — Function resolverTemplate()

## Status
Done

## Story
**As a** cidadão votante acessando a pesquisa final,
**I want** que o sistema encontre qual o formulário de auditoria qualitativa deve me apresentar (do pai local, ou raiz),
**so that** categorias irmãs sem definições especiais herdem sem estresse configurações de alto nível do gestor, mas exceções continuem usando formulários particulares.

## Acceptance Criteria

1. - [x] Utilidade `resolverTemplate(categoriaId: string): Promise<TemplateQualidade & { herdado: boolean } | null>` implantada limpa em `/src/lib/templates-qualidade/resolver.ts`.
2. - [x] A função pesquisa via Prisma o segmento (`categoriaId`). Se este contiver `templateQualidadeId` nativo (`!== null`), ele recupera o Template associado com `herdado: false` e finaliza early return.
3. - [x] Se o segmento não possuir Template nativo local, ele verifica a existência do `paiId` nele. Se possuir, faz um lookup no DB do segmento `paiId` analisando seu Template respectivo;
4. - [x] Se o segmento `paiId` possuir e retornar template, retornar este atrelado com a flag `herdado: true` com a propriedade auxiliar `herdadoDe: string_nome_segmento_pai`.
5. - [x] Permitir a varredura "Walk up" de até 3 escopos (Filho → Pai → Avô). Limitar profundidade intencionalmente no service para evitar deadlock loops.
6. - [x] Otimização no backend ou Rota: Disponibilizar a Rota Wrapper `GET /api/segmentos/[id]/template-qualidade` retornando o template resolvido.
7. - [x] A hierarquia existente via Service `segmento-service.ts` deverá em eventuais queries populacionadas poder plugar esse helper ou usar cache request memorizada para agilizar.
